## Codebase Patterns
- Global config lives in `scripts/consts.js` as `isometricModuleConfig` (MODULE_ID, DEBUG_PRINT, WORLD_ISO_FLAG, FOUNDRY_VERSION)
- Config values are initialized in the `Hooks.once("init")` handler in `scripts/main.js`, BEFORE register*() feature calls
- Version-conditional code uses `isometricModuleConfig.FOUNDRY_VERSION === 11` to branch between v11 and v13 APIs
- Foundry v11 vs v13 API differences: wall endpoints (wall.A/B vs wall.edge.a/b), visibility testing (canvas.effects.visibility vs canvas.visibility), token sorting API
- `module.json` compatibility is set to minimum v13 — v11 branches are legacy compat code
- Keep README Foundry-version badges and compatibility copy aligned with `module.json` compatibility fields to avoid mixed support signals
- No build step for JS — plain ES modules loaded directly by Foundry VTT
- All scripts are in `scripts/` directory, entry point is `scripts/main.js`
- Projection types and isometric constants are defined in `scripts/consts.js`
- Use `node --check <file>` for basic syntax validation (no tsconfig/typecheck infra yet)
- Background transform uses save/restore pattern in `transform.js` — `resetBackgroundTracking()` must be called before transforms on each new canvas (called in `handleCanvasReady`)
- Stage toggles must re-run `adjustAllTokensAndTilesForIsometric` for both enable and disable so token/tile meshes fully reset
- Use `scripts/logger.js` helpers (debugLog/debugWarn/debugGroup/debugTable) for any debug output; logs are gated by `isometricModuleConfig.DEBUG_PRINT` and prefixed with `[Isometric Perspective]`
- Expensive hook-driven layers (occlusion, dynamic tiles) should call the shared `throttle` helper (scripts/utils.js) and schedule updates instead of firing per-hook
- Normalize `linkedWallIds` through `normalizeLinkedWallIds` before read/write; only call `setFlag` when normalized output differs to avoid recursive update churn
- Core Foundry prototype patches should be registered during `init` and prefer `libWrapper` `WRAPPER` mode for interop; use guarded direct fallback only when `libWrapper` is unavailable
- Use `SMOKE-TEST.md` US-008 section as the release validation artifact: every scenario should capture Actual + PASS/FAIL before tagging
- Keep `PERFORMANCE-BASELINE.md` updated with one completed run per release candidate; treat `off` as the safe global default and `cpu6` as the recommended CPU occlusion fallback unless new measurements justify a change
